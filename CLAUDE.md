# CLAUDE.md ‚Äî AI Context

## Project Overview

Self-contained deployment script for a personal VPN server using Xray VLESS+REALITY protocol with a Flask web dashboard. Designed for censorship circumvention in regions with deep packet inspection (DPI).

## Architecture

```
Internet ‚Üí :443 (Xray VLESS+REALITY) ‚Üí VPN tunnel
              ‚Üì (via VPN loopback)
           :8080 (Flask Dashboard) ‚Äî blocked on public interface
           :10085 (Xray Stats API) ‚Äî localhost only
```

### Key insight: Dashboard access through VPN
The dashboard binds to `0.0.0.0:8080` but iptables blocks port 8080 on the public interface. VPN clients access it because Xray's `freedom` outbound connects to the server's own IP via loopback, bypassing the firewall rule. This is NOT a tunnel VPN (no tun/tap device, no VPN subnet) ‚Äî VLESS is a proxy protocol.

## Files

### `vpn-setup.sh`
The main artifact. Self-contained bash script that embeds all configs as heredocs:
- Xray config (JSON with shell variable interpolation via `${VAR}`)
- Flask app.py (written with `'APPEOF'` quoted heredoc to prevent expansion, then `sed` replaces `__PLACEHOLDER__` values)
- HTML template (quoted heredoc, no substitution)
- Systemd service, logrotate, fail2ban configs

**Dynamic values** generated at runtime:
- `SERVER_IP` and `IFACE` ‚Äî auto-detected from `ip route get 1.1.1.1`
- `PRIV_KEY` / `PUB_KEY` ‚Äî `xray x25519` keypair
- `CLIENT_UUID` ‚Äî from `/proc/sys/kernel/random/uuid`
- `SHORT_ID` ‚Äî `openssl rand -hex 8` (the *initial* shortId, kept as the global fallback; per-client shortIds are added later by the dashboard)

**Placeholder pattern**: app.py uses `__SERVER_IP__`, `__REALITY_PBK__`, `__REALITY_SID__`, `__NET_IFACE__` replaced by `sed` after heredoc write.

### `cloud-init.yml`
Generated by Python `yaml.dump()` from the raw script content. Must be regenerated (not hand-edited) when `vpn-setup.sh` changes:

```bash
python3 -c "
import yaml
with open('vpn-setup.sh') as f: script = f.read()
cfg = {'final_message': 'VPN setup complete. Check /opt/vpn-credentials.txt',
       'package_update': True, 'packages': ['curl','unzip','openssl'],
       'runcmd': ['bash /opt/vpn-setup.sh 2>&1 | tee /var/log/vpn-setup.log'],
       'write_files': [{'content': script, 'path': '/opt/vpn-setup.sh', 'permissions': '0755'}]}
with open('cloud-init.yml','w') as f:
    f.write('#cloud-config\n')
    f.write(yaml.dump(cfg, default_flow_style=False, allow_unicode=True, width=4096))
"
```

### Dashboard (embedded inside `vpn-setup.sh`)

The Flask app and HTML template are **only** inside `vpn-setup.sh` heredocs ‚Äî there is no separate `dashboard/` directory. Single source of truth.

- **app.py heredoc** ‚Äî starts at `cat > /opt/vpn-dashboard/app.py << 'APPEOF'` (search `APPEOF`). Quoted heredoc (no shell expansion); `sed` replaces `__SERVER_IP__`, `__REALITY_PBK__`, `__REALITY_SID__`, `__NET_IFACE__` after write.
- **index.html heredoc** ‚Äî starts at `cat > /opt/vpn-dashboard/templates/index.html << 'HTMLEOF'` (search `HTMLEOF`). Quoted, no substitution.
- **Chart.js** ‚Äî downloaded to `/opt/vpn-dashboard/static/chart.min.js` during install (not embedded).

**Routes:**
- `GET /` ‚Äî serves index.html
- `GET /api/system` ‚Äî CPU, RAM, disk, network, Xray status
- `GET /api/history?range=1h|6h|24h` ‚Äî time-series metrics from background collector
- `GET /api/clients` ‚Äî list clients with traffic stats from Xray Stats API
- `POST /api/clients` ‚Äî add client (writes to xray config + clients.json, generates a per-client `shortId` via `secrets.token_hex(8)` and appends it to `realitySettings.shortIds[]`, then restarts Xray). REALITY's hard limit is 8 entries in `shortIds[]` ‚Äî when reached, the new client falls back to the global `SHORT_ID` and the response sets `pooled: true`.
- `DELETE /api/clients/<id>` ‚Äî remove client and prune its shortId from `shortIds[]` if no other client uses it. The original global `SHORT_ID` is never pruned (legacy clients without a `shortId` field in `clients.json` keep working through it).
- `GET /api/clients/<id>/qr` ‚Äî QR code as SVG
- `GET /api/netflow` ‚Äî parsed access log (last 500 lines). Frontend filter (`filterNetflow()` in HTML heredoc) joins all entry fields (`time, source, status, proto, dest, route, email`) into one string and does whitespace-tokenized AND match (e.g. `tcp accepted` matches rows containing both). When adding a column to the netflow table, also extend the `hay()` join ‚Äî otherwise the new field won't be searchable.
- `GET /api/netflow/stats` ‚Äî aggregated netflow (hourly, protocols, top destinations, per-client)
- `POST /api/xray/restart` ‚Äî restart Xray service

**Important pattern ‚Äî `restart_xray_bg()`**: Xray restart is delayed 1.5s in a thread so the HTTP response reaches the client first. This matters because the VPN tunnel carries the HTTP request ‚Äî restarting Xray immediately would kill the connection before the response is sent.

**Metrics collector thread**: `metrics_collector()` samples CPU/RAM/net/conns every 30s into `deque(maxlen=2880)` (24h history), persists to `metrics_history.json` every 5min. Started as daemon thread before `app.run()`.

**Client-name transliteration**: The "new client" form whitelists `[A-Za-z0-9 _-]` and transliterates Cyrillic on the fly. Two parallel implementations of the same map exist ‚Äî `translit_name()` in `app.py` (server-side, also defends against direct curl) and `TR`/`translit()`/`translitName()` in the HTML heredoc (UX). When extending the alphabet (e.g. adding Ukrainian `—ñ/—ó/—î`), update **both** maps. Pre-existing clients with Cyrillic names are not retroactively migrated ‚Äî delete + re-add via UI to normalize.

## Common Tasks

### Adding a new API endpoint
1. Edit the embedded app.py heredoc inside `vpn-setup.sh` (search for `APPEOF`)
2. If UI needed, edit the embedded HTML heredoc (search `HTMLEOF`)
3. Add rate limiting decorator if the endpoint mutates state
4. Regenerate `cloud-init.yml`
5. Mirror the change to the live dashboard (see below) ‚Äî `vpn-setup.sh` is only consumed by cloud-init on a fresh deploy, edits there don't affect a running server

### Mirroring edits to the live dashboard
`vpn-setup.sh` is the source of truth for *new* deploys, not a runtime artifact. After editing its `APPEOF`/`HTMLEOF` heredocs, replicate the same edit in:
- `/opt/vpn-dashboard/app.py` (already has `__SERVER_IP__` etc. substituted ‚Äî don't overwrite the whole file, just edit the lines you changed)
- `/opt/vpn-dashboard/templates/index.html`

Then:
```bash
python3 -c "import ast; ast.parse(open('/opt/vpn-dashboard/app.py').read())"  # syntax check
systemctl restart vpn-dashboard && systemctl is-active vpn-dashboard
```
Hard-refresh the browser (Ctrl+F5) when changing `index.html` ‚Äî the dashboard sets no cache headers and Chrome will hold the old file.

### Changing firewall rules
Rules are in section `# ‚îÄ‚îÄ 15. Firewall` of `vpn-setup.sh`. Current whitelist:
- lo (loopback)
- ESTABLISHED/RELATED
- TCP 22 (SSH)
- TCP 443 (Xray)
- UDP 68 (DHCP)

### Changing security headers
In `@app.after_request` decorator in `app.py`. CSP requires `'unsafe-inline'` for script-src and style-src because the dashboard uses inline script/style blocks.

## Testing

After running `vpn-setup.sh`:
```bash
systemctl is-active xray           # should be "active"
systemctl is-active vpn-dashboard  # should be "active"
fail2ban-client status sshd        # should show jail active
curl -I http://127.0.0.1:8080/     # should show security headers
cat /opt/vpn-credentials.txt       # VLESS link for client
```

## Gotchas

- **REALITY key format**: `xray x25519` output labels vary between versions ‚Äî the grep for `PrivateKey` and `Password` (not `PublicKey`) matches the current Xray output format
- **Xray has no log-reopen signal**: That's why logrotate uses `copytruncate` instead of the usual rename+SIGHUP pattern
- **cloud-init.yml must be regenerated, not hand-edited**: YAML escaping of the bash script is non-trivial (quotes, backslashes, dollar signs)
- **NET_IFACE is not always `ens1`**: The script auto-detects it and injects via `sed` (`__NET_IFACE__` placeholder)
- **Xray config mode 644, not 600**: Xray runs as `nobody` (see `User=nobody` in systemd unit); config needs world-readable permissions
- **DNS leak protection** lives in two places: section `6b` configures systemd-resolved with DoT (system-level), and section `7` (Xray config heredoc) includes Xray's DoH DNS module (Cloudflare + Google) plus `domainStrategy: UseIPv4` on the `freedom` outbound so domains resolve via Xray's DNS, not the OS resolver
- **Per-client shortIds**: Dashboard generates a fresh `shortId` for each new client and appends it to `realitySettings.shortIds[]` (max 8 ‚Äî REALITY hard limit). Each client's `clients.json` entry stores `shortId` so QR/link generation picks the right one. Clients that predate this feature have no `shortId` field and use the global fallback (`REALITY_SID` constant in app.py, set from `SHORT_ID` at install). When migrating an old client to a per-client sid, delete and re-add it via the dashboard
- **Fail2ban uses nftables, not iptables**: Rules live in table `inet f2b-table` (inspect with `nft list table inet f2b-table`), NOT visible via `iptables -L`. The main firewall (section 15) is still iptables ‚Äî two separate subsystems coexist. To override ban behavior (e.g. drop vs reject), create `/etc/fail2ban/action.d/nftables.local` with `[Init]` / `blocktype = drop` ‚Äî never edit `nftables.conf` directly (package updates overwrite it)
